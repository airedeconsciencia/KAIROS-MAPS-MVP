════════════════════════════════════════════════════════
NOTA DE CORRECCIÓN v2.0
════════════════════════════════════════════════════════


La versión anterior (v1.0) definía todos los cálculos complejos
en Cloud Functions / backend. Esto es la arquitectura IDEAL a largo plazo,
pero es incompatible con el motor actual de KAIROS, que funciona
100% en el cliente (JavaScript + Swiss Ephemeris WASM).


DECISIÓN ARQUITECTURAL:
  Fase 1 — MVP: Todo el cálculo en cliente (JS + WASM)
  Fase 2 — Escalado: Migrar cálculos a Cloud Functions


Esta versión corregida refleja la Fase 1 correctamente.
El documento original permanece en la carpeta como referencia para Fase 2.

0. PRINCIPIO BASE DEL SISTEMA (FASE 1 — MVP)
════════════════════════════════════════════════════════


Regla fundamental para el MVP:


  CALCULAR en el cliente (JS + WASM)
  GUARDAR en Firestore solo lo necesario
  CACHEAR resultados pesados para no recalcular
  MANTENER la estructura lista para migrar a backend en Fase 2

│ Tipo                  │ Fase 1 (MVP)  │ Fase 2      │

0.1 PRIVACIDAD Y MINIMIZACIÓN DE DATOS
════════════════════════════════════════════════════════


Kairos Maps trabaja con datos sensibles:
  - fecha de nacimiento
  - ubicación
  - patrones emocionales
  - objetivos vitales


La privacidad debe estar integrada desde el diseño.
La app debe:
  - pedir permisos claros
  - permitir borrar todos los datos
  - permitir desactivar localización en cualquier momento
  - NO almacenar historial GPS continuo
  - minimizar almacenamiento de datos emocionales

13. QUÉ SE CALCULA DÓNDE — FASE 1 vs FASE 2
════════════════════════════════════════════════════════


FASE 1 — MVP (cliente JS):
  ✓ Posiciones planetarias (planetary_engine.js + Swiss Ephemeris WASM)
  ✓ Ascendente y ángulos (ascendant_engine.js)
  ✓ Líneas astrocartográficas (astrocarto_engine.js — nuevo)
  ✓ Ángulos relocados (astrocarto_engine.js)
  ✓ Aspectos (aspects_engine.js)
  ✓ Scores de lugar (lógica JS)
  ✓ Interpretación IA (llamada a API Anthropic desde cliente)
  ✓ Render del mapa (Leaflet.js)


  Se guarda en Firestore:
  ✓ Datos natales
  ✓ Carta natal calculada (cache)
  ✓ Preferencias
  ✓ Análisis de lugares (resultado final)
  ✓ Lugares guardados
  ✓ Memoria interpretativa


FASE 2 — Escalado (Cloud Functions):
  → Los cálculos pesados migran a Firebase Functions
  → La estructura de Firestore no cambia (esto es clave:
     diseñar bien Firestore en Fase 1 evita migraciones costosas)


════════════════════════════════════════════════════════
14. FLUJO COMPLETO — FASE 1
════════════════════════════════════════════════════════


1. Usuario se registra → Firestore crea users/{userId}
2. Introduce datos natales → guardados en birthData/main
3. Cliente calcula carta natal con planetary_engine.js
4. Carta natal cacheada en charts/natal (no recalcula en cada sesión)
5. Cliente calcula líneas astrocartográficas con astrocarto_engine.js
6. Líneas guardadas en localStorage (JSON pesado)
7. Resumen de líneas guardado en astroMaps/main (Firestore)
8. Usuario busca un lugar
9. Cliente calcula análisis del lugar:
   - getLinesNearCity() → líneas cercanas
   - getRelocatedAngles() → ángulos relocados
   - Lógica de scores
10. Cliente llama a API Anthropic con el contexto estructurado
11. IA genera interpretación personalizada
12. Resultado guardado en placeAnalyses/{analysisId}
13. Usuario puede comparar lugares, guardar favoritos, ver historial


════════════════════════════════════════════════════════
15. ERROR CRÍTICO A EVITAR
════════════════════════════════════════════════════════


NO guardar el análisis completo como texto plano:


MAL:
{ "analysis": "Texto larguísimo generado por la IA..." }


BIEN:
{
  "scores": { "amor": 8.4, "trabajo": 6.1 },
  "dominantEnergy": "venus_dc",
  "warnings": ["idealización"],
  "aiInterpretation": { "shortSummary": "...", "longAnalysis": "..." }
}


Por qué: los scores y la estructura son estables y consultables.
El texto de la IA puede regenerarse si mejora el modelo.
Los datos estructurados permiten comparar y rankear ciudades.
